Technical Approach Selection

Choose the architecture for your voice-based mock interview app

A

Pipeline Assembler

Deepgram STT + Vercel AI SDK + ElevenLabs TTS

Discrete best-in-class services assembled via WebSocket. Single Next.js monolith, TypeScript only.

Pros

  • Single language (TypeScript)
  • Simple deployment (Vercel)
  • Full multi-model parity
  • Low vendor lock-in
  • MVP: 4-6 weeks

Cons

  • Latency ~1.0-1.3s (3 network hops)
  • Interruption handling is manual
  • Hard to add video later

Cost/interview: ~$1-2.50

B

Native Realtime (OpenAI)

OpenAI Realtime API via WebRTC (speech-to-speech)

Single API handles STT+LLM+TTS. Lowest latency for OpenAI models, but needs fallback pipeline for others.

Pros

  • Best latency ~400-600ms
  • Simplest primary path
  • Built-in interruption handling
  • MVP: 3-5 weeks

Cons

  • High vendor lock-in (OpenAI)
  • Non-OpenAI models get worse UX
  • Dual pipeline = double complexity
  • Cost: ~$3.60+/interview

Cost/interview: ~$3.60+

C

LiveKit Orchestrator

LiveKit Agents (Python) + WebRTC SFU

Open-source voice infra. Python agent handles STT/LLM/TTS pipeline with production-grade turn-taking and interruption.

Pros

  • Best multi-model support
  • Built-in interruption handling
  • Native video support (future)
  • Low cost, low lock-in
  • Latency ~750-950ms

Cons

  • Two languages (TS + Python)
  • Two services to deploy
  • Most complex architecture
  • MVP: 5-7 weeks

Cost/interview: ~$1-2.50

Recommendation

Best long-term product: C (LiveKit) -- production-grade voice infra, true multi-model parity, future video support

Fastest to ship: A (Pipeline) -- single language, simple deployment, good enough latency for MVP

Best latency (OpenAI only): B (Realtime) -- but high cost and vendor lock-in make it risky as foundation
